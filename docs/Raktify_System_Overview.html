01 · Executive summary
In one page

A permanent system,
not a one-time camp.

India donates ~12.5 million units of blood a year. The country still falls 1 million units short of demand, and almost every shortfall is solved by frantic WhatsApp messages, phone trees and trust. Raktify replaces that improvised infrastructure with software — carefully designed for the conditions Indian healthcare actually operates in.

What the platform does

  • Lets any donor register on mobile in under 90 seconds, in Marathi / Hindi / English, with full consent.
  • Lets any hospital raise an emergency request in a single screen — patient + units + urgency + needed-by — and matches against verified bank inventory in seconds.
  • Lets blood banks run their full bag lifecycle — donations recorded, TTI tested with 4-eyes verification, inventory tracked, recalls cascaded.
  • Lets NGO coordinators orchestrate the response district by district, with ring-escalation when supply doesn't materialise.
  • Lets anyone — a Rotary club, school, panchayat, corporate — host a donation camp through a public application, with NGO verification and a magic-link organiser dashboard.

What makes Raktify different from a donor directory

End-to-end bag lifecycle
From donor consent to TTI clearance to bag issuance and post-transfusion confirmation — every step audited, every state legal.
Privacy is a feature
Donor mobile numbers never reach hospitals. TTI results never leave the screening tab. AES-256-GCM encryption. Hash-chained audit log.
Built for Indian conditions
Mobile OTP first. Offline-capable PWA. Marathi-default. Free always. No app-store dependency.
Component-level matching
Asks "PRBC for B+ patient, fallback to O+ allowed, expiry > 8 days, recall-free" — not "have any B+ blood?"
One sentence to remember. Raktify is not a donor directory; it's a life-critical operating system for the moment when someone with a rare blood type, in a small district, at 2 a.m., needs to find the one available unit and have it in the operating theatre within two hours.

02 · The problem we're solving
Why this matters

A million units short.
And the gap is closed by panic.

India has world-class hospitals and a culture of voluntary giving. What it lacks is plumbing — a quiet, reliable layer that turns "we need a unit of B-negative for a bleeding mother in a sub-district hospital" into "here is the verified unit, in this fridge, with this bag ID, dispatched at this time."

~12.5M
units collected per year in India
~1M
annual shortfall — about 1 in every 13 requests
35%
of maternal deaths in India involve haemorrhage where timely blood matters

The shortfall is solved by improvisation

  • A patient’s family forwards a "B-negative urgently needed" WhatsApp message to dozens of strangers.
  • Coordinators run paper rosters and personal contact books that no one else can audit, update, or inherit.
  • Blood banks across districts have no way to see each other’s inventory; surplus expires in one fridge while a hospital 50 km away dispatches a frantic search.
  • Rare phenotype donors (Bombay group, Rh-null) are tracked on the chairman’s phone. When the chairman is unreachable, so is the donor.
  • Donors receive no feedback after donating — no proof, no impact, no tier recognition. Most drop off after their first donation.
From a real case in Amravati, March 2025. A six-year-old with thalassemia needed her monthly transfusion. The local blood bank's stock had expired. A neighbouring district had stock — but no one knew. By the time a Rotary volunteer phoned around, the child had skipped a cycle. Repeat this 50,000 times a year, across India.

Existing platforms are donor directories, not operating systems

ublood, friends2support and similar tools answer "where is a B-positive donor?" That is a useful but incomplete question. What hospitals and blood banks need is the operational depth that turns a verified donor into a verified unit, on time.

04 · An emergency in motion
The 10-minute story

A B-negative request,
handled end-to-end.

To make the system tangible, here is one emergency from request to closure. Every step happens inside Raktify; every step is logged in the audit chain; every step respects the privacy invariants.

1
10:42 — Hospital raises the request
Irwin Hospital's casualty officer opens the Raise new request tab. Patient initials, age, gender, blood group B-, component PRBC, 2 units, urgency Critical, needed by 14:00. Clinical indication: postpartum haemorrhage. Submit.
2
10:42 — Matching engine fires
The engine pulls the compatibility matrix (B- patients can receive B- or O-), filters Amravati district blood banks for fresh, non-recalled PRBC bags, sorts by expiry FIFO, and reserves the two best bags at Amravati Blood Centre. Response returned to the hospital in under a second.
3
10:43 — Coordinator on duty is notified
Anjali, the Amravati district coordinator, sees the request appear in her queue (auto-refreshes every 15 s). The matched blood bank is shown. The cross-role thread is open for negotiation. The patient's mobile number is never exposed to the hospital or the coordinator.
4
11:02 — Blood bank issues the bags
Amravati Blood Centre confirms the reservation in their portal and the bags move from AV Available to IS Issued. The hospital is shown the bag IDs.
5
13:18 — Crossmatch confirmed by the hospital
Post-transfusion, the hospital taps Confirm crossmatch. The request flips to CL Closed. The bags move from IS → TR Transfused. Time-to-fulfilment: 2 hours 36 minutes — recorded as a KPI on the hospital dashboard.
6
Audit + outcome
Every state change between steps 1 and 5 is captured in the hash-chained audit_log. The donor who originally gave each bag sees their donation tier tick up. The hemovigilance report aggregates this case in next month's DHO submission.
The whole flow took six clicks and was visible to everyone authorised to see it. What used to take a Rotary chairman 90 minutes of phone calls now takes Raktify under three hours, with full traceability. The improvement isn't speed alone — it's that nobody dies because a chairman lost his phone.

Stakeholder 01 — The Donor

Ramesh, 34, civil engineer, Amravati. O+. Wants to help, hates spam.

A regular Indian who donates blood occasionally. Carries a mid-range smartphone, uses WhatsApp constantly, reads in Marathi by default, dislikes installing apps, and trusts an organisation more when it asks for less.

What Ramesh actually does

1
Signs up in 90 seconds
Opens raktify.choudhari.ngo from a WhatsApp link. 4-step wizard: pre-screening questions → name + DOB + village → temporary-deferral notice → consent + OTP. No app, no install, no email.
2
Sees one big toggle: "Mark me available today"
The donor dashboard has exactly one important control. He taps it green on weekends, leaves it grey when travelling. Works offline — the change syncs when he’s back online.
3
Gets WhatsApp'd about a nearby camp
Rotary Amravati hosts a camp on Saturday. Ramesh receives a WhatsApp message with a one-tap link, opens the public camp page, taps "I will be there." His RSVP shows up on the organiser’s roster — but his phone number is never shared.
4
Donates. Gets a tier badge.
After the camp, the blood bank records his donation. His donor dashboard shows a Bronze badge (1 donation) with progress towards Silver. An estimated-lives-saved counter ticks up. He shares the badge with friends. Three of them sign up that week.
5
Gets recalled. Quietly.
Three months later, his donation’s TTI re-test surfaces a borderline reading. The platform recalls the bag, schedules him for re-testing through the hospital — never blasted publicly, never on a WhatsApp group, never named.

What Ramesh sees that no other platform shows him

  • Verified blood group, lab-confirmed (not "self-reported")
  • Next-eligible date, computed from his last donation + component shelf life
  • TTI clearance verdict per donation (verdict only — no field-level results)
  • Marathi by default — switches to Hindi or English on tap

What Ramesh never sees, by design

  • Patient names or hospital identities behind a request
  • Other donors' phone numbers, even from his own community
  • Field-level TTI test results that could be misread by a non-clinician
  • Any nag for money — the platform is free to him, always

Stakeholder 02 — The Hospital

Dr. Deshmukh, Medical Superintendent, Irwin Hospital, Amravati.

Runs a 200-bed government hospital with one casualty officer per shift. Has seen patients die because the blood bank didn’t have the right unit and the phone trees were too slow. Wants something that just works, and that their nurses can use in seconds.

Hospital fills its legal name, shortname, CDSCO licence (if it operates a bank), Clinical Establishments Act number, location, primary contact. Submits.

NGO verifies + sends the MoU for eSign
The Choudhari Foundation verifies the licence (manually for now — automated check is on the roadmap), then sends an Aadhaar eSign request via LeegAlly to the primary contact’s WhatsApp.

eSign completes → admin login provisioned automatically
The eSign webhook fires, the institutional admin account is auto-created, and credentials are WhatsApp’d to the signatory. From signup to login, no human intervention required after the NGO clicks "Send MoU".

What the hospital portal looks like daily

Dashboard
90-day KPIs (open / critical / fulfilled / expired / avg time-to-fulfilment), district inventory grid (counts, not bag IDs), recent activity strip.
My requests
Per-request card: urgency pill, status, fulfilled / required counts, raised time, matched blood bank, coordinator. One-tap Confirm crossmatch to close.
Raise new request
One screen. Validates client-side. Backend runs the matching engine synchronously and returns matched bag count + fallback flag in the response.

07 · The blood bank's journey
Stakeholder 03 — The Blood Bank

Dr. Kale, Director, Amravati Blood Centre. CDSCO-licensed.

Runs a district blood centre that processes ~60 donations a week, separates them into components, and supplies four hospitals. Today, her staff handle inventory on paper logs, dispatch by phone, and reconcile at month-end. She wants software that her counter staff can use without a one-week training course.

The four tabs of the blood-bank portal

Dashboard

At a glance, every shift
5 KPI cards (Available units, Expiring <48h, Pending TTI, Issued this month, Donations today). Inventory grid (8 groups × N components). Incoming requests in the district — the "Raise Hand" panel for matching.

Inventory

Bag-level, expiry colour-coded
Every bag has an ISBT barcode, blood group, component, volume, status (QA → AV → RE → IS → TR / EX / RC). Expiry <48h is shown red, 2–7 days amber, >7 days green. Single-tap recall.

Record donation

Mobile lookup → pre-filled form
Counter staff types the donor’s mobile, the platform auto-fills their verified blood group + deferral state. Submit creates a QA bag automatically via trigger.

TTI screening

5 panels, 4-eyes verify
HIV / HBsAg / HCV / Syphilis / Malaria entered as NR / RR / PE / ID. A different staff member must verify. Reactive results cascade to bag recall and a lookback investigation automatically.

Why this is hard to build, and why we built it anyway

  • The 4-eyes rule for TTI is enforced by a database trigger — not a UI checkbox. The screening can’t be verified by the same user who entered it. Application code has bugs; constraints don’t.
  • Component separation is modelled per-bag (PRBC, FFP, SDP, PLT, CRY, WB) with the correct shelf life and storage temperature per component. The matching engine respects this.
  • Recalls are not a status change — they're an event. Every other related row (lookback registry entry, donor alert, hospital notification) cascades automatically.
The CDSCO-grade promise. Raktify can’t replace a blood bank’s lab work, but it gives the bank an auditable digital twin of every bag it touches — searchable, reportable, recallable, and DHO-submission-ready.

The Choudhari Foundation operations team.

The trust gate. Verifies every institution that wants to join, vets every coordinator, reviews every camp application, monitors lookback investigations, files the DHO hemovigilance report monthly, and audits the audit log itself.

Ten tabs, one console

Institution applications. Verify licence → send MoU eSign → activate. Default landing tab.

Coordinators

Verify, suspend with reason, monitor reliability scores district-wide.

Camps

Public camp-host applications. Review, approve (mint magic link), decline with reason.

Thalassemia

Recurring-transfusion patient registry with due-soon highlighting. Record transfusion in one tap.

Rare blood

Bombay, Rh-null, weak-D donors. Both donor-linked and shadow entries supported.

Duplicates

Suspected duplicate donor pairs. Clear false positives. Merge (pending medical sign-off).

Referrals

Funnel from donor-referred-an-institution → onboarded. Conversion rate, recent referrals.

Lookback

Open post-reactive-TTI investigations. Red highlight when older than 14 days.

Audit

Filterable audit log + hash-chain integrity check. Detects any backdating or tampering.
The audit chain is non-negotiable. Every state change in Raktify — every consent flip, every TTI verification, every bag recall — is captured in a cryptographic hash chain. Even a future super-admin can't backdate a row without breaking the chain. The Audit tab’s integrity check tells the auditor whether the chain is intact in seconds.

Reports the admin can produce

  • District summary — requests raised / fulfilled / expired, avg response, shortages by group, donor pool, camps, wastage.
  • Hemovigilance — DHO-eligible. Lookback opened / closed, reactive TTI counts, adverse reactions, donation source breakdown.
  • Blood bank performance — inventory accuracy, fulfilment counts, avg TTI entry latency.

Reports come out as

  • JSON for partners who want to ingest the data
  • CSV for DHO submissions today
  • PDF generation (Puppeteer) on the Q4 2026 roadmap

12 · Privacy & safety
Non-negotiable

Privacy is a load-bearing pillar,
not a checkbox.

A life-critical platform that handles encrypted health data, mobile numbers, and screening results needs to be defensible against curious admins, leaked database connections, malicious app servers, and a future hostile audit. Raktify is engineered to meet that bar.

Row-Level Security
Every table has RLS policies keyed off the actor’s role and identity. A coordinator can’t read another district’s data, even if they have the database password.
Hybrid encryption
PII columns are AES-256-GCM encrypted with two distinct KMS keys — main + screening. A compromised app server can’t read TTI data without separately compromising the screening key.
Hash-chained audit
audit_log is INSERT-only. Every row contains a SHA-256 hash of the previous row's salient fields. Backdating a record breaks the chain — the integrity check detects it in seconds.
Mobile masking
Donor mobile numbers are encrypted at rest, never returned to hospitals or camp organisers, and shown only to admins as +91XXXXX1234.
TTI never leaks
Field-level TTI results never leave the screening tab. Donors and coordinators see only the verdict: Pending / Cleared / Investigation / Held.
Self-reported ≠ verified
Donor self-reported blood group is rendered with an "Unverified" badge and is never queried by the matching engine. Lab-verified only.

The clinical safety rules live in the database

Application code has bugs. Database constraints, triggers, and CHECK clauses don’t. That’s why every patient-safety rule lives at the database layer:

  • 4-eyes TTI verification is enforced by a trigger that rejects any verification where the verifier and the entry user match.
  • Self-reported blood group can never be promoted to the matching column without going through the blood-bank-only verification endpoint.
  • Consent grants must be self-initiated by the donor — a trigger blocks any other actor from setting consent_data_use=TRUE.
  • Audit log is INSERT-only at the role level — no application role has UPDATE or DELETE.
Compliance posture. Aligned with DPDP Act 2023 (data-fiduciary obligations), CSR Schedule VII (i) preventive healthcare & (xii) disaster management. Data residency in Azure Central India. Subject-access export endpoint on the roadmap.

13 · Technical architecture
For the technically curious

Boring tech. Carefully composed.

We chose well-understood components over novelty. Postgres over a NoSQL store. Server-rendered HTML for crawlers and SPA-style React for users. Provider abstractions for every external integration so we’re never locked in.

The stack

Database PostgreSQL 17 with Row-Level Security on every table Azure PG Flexible Server · B1ms · Central India · Multi-AZ ready
Backend API Node.js 22 LTS · Express · pg · Zod validators · pino structured logs Azure App Service · Linux B1 · Oryx build
Frontend React 18 · Vite 5 · Tailwind 3 · React Query 5 · PWA with offline outbox Azure Static Web Apps · Free tier · Cloudflare DNS
Auth — donors Mobile OTP, JWT, mobile-OTP-only login WhatsApp Cloud (Meta-direct) / MSG91 / console
Auth — staff Email + bcrypt + TOTP for institutional admins Rate-limited 10/15min per IP
Encryption AES-256-GCM, two-key hybrid (main + screening) Local key in dev · Azure Key Vault / AWS KMS in prod
File storage Storage provider abstraction Local disk (dev) · Azure Blob (prod) — same API
Digital MoU LeegAlly Aadhaar eSign with provider stub for local dev Webhook-driven auto-provisioning
Geo data LGD (Local Government Directory) bulk importer API or CSV mode · idempotent re-imports
CI/CD GitHub Actions with Azure OIDC federated identity Backend + Frontend deploy on push to main
Observability Pino structured logs · planned Sentry + Application Insights Health check at /health · DB ok flag

Three architectural commitments

Patient safety in the DB
CHECK constraints, triggers and RLS — not application code. Apps have bugs; constraints don’t.
Provider abstractions
Notifications, encryption, storage, eSign are all swappable behind a one-env-var flag. No vendor lock-in.
Immutable migrations
Every migration has a SHA-256 checksum + rollback block. The runner refuses to re-apply a changed migration.
